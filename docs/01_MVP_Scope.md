# MVP 범위

## 필수 구현

| 영역 | 내용 | 목적 |
| --- | --- | --- |
| A기업 햄버거 주문 | 실제 주문 API와 DB 저장 | 발표에서 실제 처리 결과 확인 |
| B/C기업 Mock 키오스크 | 서로 다른 UI 구조 | 기업별 UI 차이 문제 제기 |
| Dajeong Text Chat | 자연어 주문 후보 생성과 승인 | 통합 인터페이스 시연 |
| Burger MCP Adapter/Server | 메뉴 조회, 최근 주문 조회, 주문 생성 tool 경계 | MCP 기반 구조 시연 |
| 관리자 페이지 | 주문 목록, 주문 상세, 상태 변경, MCP 로그 | 실제 처리 결과 검증 |
| 로그인/회원가입 | 사용자와 관리자 계정 | 기본 시연 흐름 |
| Seed 데이터 | 메뉴, 사용자, 최근 주문, 선호/비선호 재료 | 발표 안정성 |
| Mock 결제 | 결제 상태 저장 | 주문 완료 흐름 표현 |

## 선택 구현

| 영역 | 내용 | 적용 조건 |
| --- | --- | --- |
| Toss 테스트 결제 | 테스트 결제창 또는 adapter | 필수 흐름 완성 후 |
| Korail 시간표 조회 | 시간표 조회와 예약 후보 mock | 햄버거 주문 흐름 완성 후 |
| 실제 LLM Provider | `LLMProvider` 교체 구조 | 규칙 기반 demo가 안정화된 후 |
| Voice 선택 UI | Text/Voice 선택 화면 | Voice 기능은 비활성 상태로 표시 |

## 제외 범위

- 실제 Toss 실결제
- 실제 카드번호 또는 민감정보 저장
- 실제 코레일 좌석 예약/발권
- 비공식 자동화 또는 스크래핑 기반 예약 처리
- 실제 STT/TTS Voice
- Vector DB/RAG
- 실제 생체인식
- B/C기업 Mock 키오스크의 실제 주문 DB 연동

## 범위 판단

최소 발표 가능 버전은 Phase 0부터 Phase 6까지입니다. Toss, Korail, Voice, 실제 LLM Provider는 핵심 흐름을 보강하는 확장 기능이며, 초기 MVP 필수 조건으로 두지 않습니다.

# 발표 시연 흐름

## 핵심 햄버거 주문 시나리오

1. B/C기업 Mock 키오스크를 먼저 보여주며 같은 주문도 화면 구조가 다르다는 문제를 설명합니다.
2. A기업 키오스크에서 메뉴와 옵션을 확인하고, 실제 주문 생성이 가능한 화면임을 보여줍니다.
3. 사용자가 다정에 `user1/user1234`로 로그인합니다.
4. 사용자가 다음 문장을 입력합니다.

```text
늘 먹던 햄버거 하나 주문해줘. 오이는 빼줘.
```

5. 다정이 최근 주문, 메뉴, 제거 가능한 재료, 사용자 선호/비선호 정보를 조회합니다.
6. 다정이 주문 후보를 생성하고 사용자에게 확인을 요청합니다.
7. 사용자가 주문안을 승인합니다.
8. Burger MCP Adapter/Server가 A기업 주문 API 또는 DB에 주문 생성을 요청합니다.
9. 결제는 Mock 상태로 승인 처리합니다.
10. 사용자 화면에 주문번호, 대기번호, 결제 상태를 표시합니다.
11. 관리자 계정 `admin/dajeong`으로 관리자 페이지에 접근합니다.
12. 관리자 페이지에서 주문 상세와 MCP 호출 로그를 확인합니다.

## 발표에서 강조할 증거

- A기업 주문은 실제 저장 데이터입니다.
- B/C기업 키오스크는 문제 제기용 Mock이며 실제 주문을 만들지 않습니다.
- MCP 로그에는 tool 이름, 요청 payload, 응답 payload, 성공/실패 상태, 호출 시간이 남습니다.
- 관리자 화면은 이 프로젝트가 단순 챗봇이 아니라 tool 호출 기반 플랫폼임을 보여주는 검증 화면입니다.

## 확장 시나리오

햄버거 주문 흐름이 안정화된 뒤 아래 시나리오를 추가할 수 있습니다.

```text
내일 아침 대전에서 서울 가는 KTX 예약해줘.
```

확장 시나리오는 실제 발권이 아니라 시간표 조회, 예약 후보 생성, Korail 화면 이동 또는 mock 확정까지만 다룹니다.
